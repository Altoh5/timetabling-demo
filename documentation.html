1. Overview

The University Timetable Scheduler is a browser-based application that automatically generates conflict-free class schedules for a university department. It uses a two-phase optimization algorithm — greedy placement followed by local search — to produce high-quality timetables that respect hard constraints and optimize soft preferences.

The system is designed for demo scale: a single department with 14 courses, 6 rooms, and 6 lecturers scheduling across a Monday–Friday week from 8 AM to 6 PM.

Key design goal: A fully self-contained single HTML file with no external dependencies. Runs entirely in the browser — no server, no database, no build tools.

2. Features

Automatic Scheduling

One-click generation of conflict-free timetables using greedy placement + local search optimization.

Three Timetable Views

Weekly Grid (all classes), By Room (single room's schedule), By Lecturer (single lecturer's schedule).

Full CRUD Management

Add, edit, and delete rooms, lecturers, and courses through the sidebar forms.

Constraint Tuning

Adjustable weight sliders (1–10) for soft constraints — room fit, lecturer preference, spread, and no back-to-back.

Interactive Course Selection

Click any course block to highlight all its sessions and view a detail panel with schedule info and scores.

Quality Metrics

Displays total quality score, room utilization percentage, and warnings for unschedulable courses.

Hover Tooltips

Hover over any class block to see course name, room details, lecturer, enrollment, and soft score.

Color-Coded Legend

Each course gets a unique color. A legend below the timetable maps colors to course names.

4. Scheduling Algorithm

Phase 1: Greedy Placement

Courses are sorted by scheduling difficulty (most constrained first), then placed one session at a time into the best available slot.

Courses sorted by difficulty │ ┌────▼────┐ │ Lab │ (fewest compatible rooms) │ Seminar │ (medium constraints) │ Lecture │ (most flexible) └────┬────┘ │ For each course, for each session: │ ┌────▼────────────────────────┐ │ Generate all valid │ │ (room, day, hour) combos │ │ that pass hard constraints │ └────┬────────────────────────┘ │ ┌────▼────────────────────────┐ │ Score each combo against │ │ soft constraints │ │ Pick highest-scoring slot │ └────┬────────────────────────┘ │ ┌────▼────────────────────────┐ │ Place class in schedule │ │ (or flag as unscheduled) │ └─────────────────────────────┘

Sorting priority:

  1. Room type — labs first (fewest rooms), then seminars, then lectures
  2. Enrollment size — larger classes first (fewer rooms can fit them)
  3. Sessions per week — courses needing more sessions are harder to fit

Phase 2: Local Search

After the greedy phase, the algorithm runs up to 100 iterations of local search to improve the soft constraint score:

  1. Find the scheduled entry with the lowest soft score
  2. Temporarily remove it from the schedule
  3. Try all valid alternative (day, room, hour) placements
  4. If a better placement exists, swap it in
  5. Stop when no improvement is found or iteration limit reached
Performance: The algorithm runs near-instantly for demo scale (~14 courses, 5 days, 10 hours). Each greedy placement evaluates at most 5 days × 6 rooms × 10 hours = 300 candidate slots per session. Local search adds at most 100 × 300 evaluations. Total: well under 100,000 operations.

5. Constraint System

Hard Constraints (always enforced)

These are never violated. If no valid slot satisfies all hard constraints, the session is flagged as unscheduled.

ConstraintDescription
No room double-bookingTwo classes cannot occupy the same room at overlapping times. Uses interval overlap check: start1 < end2 AND start2 < end1
No lecturer double-bookingA lecturer cannot teach two classes at overlapping times
Room type matchA course requiring a "lab" room can only be placed in a lab-type room
Room capacityRoom capacity must be ≥ the number of enrolled students
Schedule boundsClass must fit within 8 AM – 6 PM (start + duration ≤ 18)
Lecturer max hours/dayA lecturer's total teaching hours on a single day cannot exceed their configured maximum

Soft Constraints (optimized via scoring)

Each soft constraint has a configurable weight (1–10) that controls how much it influences placement decisions. Higher weight = stronger preference.

ConstraintWeight SliderScoring Formula
Room Fit Room Fit (default: 5) weight × (1 - (capacity - enrolled) / capacity)
Penalizes oversized rooms. Perfect fit = full score.
Lecturer Preference Lecturer Pref (default: 5) +weight if the entire session falls within the lecturer's preferred time window, 0 otherwise.
Spread Spread (default: 8) -weight × count where count = number of same-course sessions already on that day. Distributes sessions across the week.
No Back-to-Back No Back-to-Back (default: 5) -weight for each adjacent session by the same lecturer on the same day. Gives lecturers breaks between classes.
Trade-off exploration: Adjust the weight sliders and regenerate to see how different priorities produce different timetables. For example, maxing "Spread" distributes sessions evenly across all 5 days, while maxing "Room Fit" minimizes wasted seats.

7. Interactions

Course Block Click

Clicking any course block on the timetable grid triggers the following:

  1. All blocks for that course get highlighted with a white border and glow
  2. All other course blocks are dimmed to 30% opacity
  3. A detail panel slides in from the right showing:
    • Course color swatch and name
    • Assigned lecturer
    • Enrollment count
    • Required room type
    • Sessions per week
    • Average soft score
    • List of all scheduled sessions (day, time, room)
  4. Clicking the same block again, clicking elsewhere, or clicking the close button deselects

Hover Tooltips

Hovering over any class block displays a tooltip with course name, room name and capacity, lecturer name, enrollment count, and individual session soft score.

Constraint Weight Tuning

The Settings section at the bottom of the sidebar provides four sliders (1–10). After adjusting weights, click "Generate Timetable" again to see a new schedule reflecting the updated priorities.

8. Design Considerations

Why Greedy + Local Search?

Three algorithmic approaches were evaluated:

ApproachProsCons
Greedy + Local Search Fast, deterministic, easy to debug, good results at demo scale May miss global optimum on very large inputs
CSP with Backtracking Guarantees finding a solution if one exists Exponential worst case, harder to incorporate soft constraints
Genetic Algorithm Can explore wider solution space Non-deterministic, slow convergence, over-engineered for demo scale

Greedy + Local Search was chosen because it runs near-instantly at demo scale, produces good-quality results, and keeps the code simple and maintainable.

Why a Single HTML File?

  • Zero deployment friction — Open the file in any browser, or host on any static server (GitHub Pages, S3)
  • No build tools — No npm, webpack, or bundler setup required
  • Fully portable — Email it, put it on a USB drive, or share a link
  • Self-contained — No CDN dependencies that could break or go offline

Why In-Memory State Only?

  • Demo-scale application does not need persistence
  • Avoids complexity of localStorage serialization or backend databases
  • Sample data reloads on every page load for consistent demo experience
  • If persistence were needed, localStorage could be added with minimal changes

Most-Constrained-First Ordering

Lab courses are scheduled first because they have the fewest compatible rooms. This prevents situations where easy-to-schedule lecture courses consume lab timeslots, leaving lab courses with no options. This is a well-known heuristic in constraint satisfaction called the "most constrained variable" or "fail-first" principle.

Overlap Detection Formula

The system uses the standard interval overlap check: two time intervals [start1, end1) and [start2, end2) overlap if and only if start1 < end2 AND start2 < end1. This cleanly handles both partial and full overlaps, as well as multi-hour sessions.

XSS Protection

All user-provided text (room names, lecturer names, course names) is sanitized through escapeHtml() before being inserted into the DOM. This prevents cross-site scripting attacks from maliciously crafted input.

Referential Integrity

When a lecturer is deleted, all courses assigned to that lecturer have their lecturerId set to null (displayed as "Unassigned") rather than keeping a stale reference. This prevents crashes during scheduling when the algorithm tries to look up a deleted lecturer.

Default Spread Weight

The spread weight defaults to 8 (higher than other weights at 5) to ensure classes distribute across all five weekdays. Without this bias, the greedy algorithm tends to cluster sessions on Monday–Wednesday due to the natural iteration order.

10. Technical Details

Technology Stack

  • HTML5 — Semantic markup, form elements
  • CSS3 — CSS Grid layout, CSS custom properties (variables), transitions, animations, pseudo-element tooltips
  • Vanilla JavaScript — No frameworks or libraries
  • Inline SVG — SIT logo rendered as embedded SVG

Browser Compatibility

Requires a modern browser with support for CSS Grid, CSS custom properties, and ES6+ JavaScript. Tested on Chrome, Firefox, Safari, and Edge.
